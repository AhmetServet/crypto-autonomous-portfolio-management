# Prediction Journal LLD

## 1. Purpose
This document describes the low-level design for a prediction journal in `src/capm`.

The prediction journal is the durable audit log for model outputs. It records each runtime prediction, then later settles that prediction when the target candle is available. This gives the project a live-quality feedback loop before and during Binance demo/testnet trading.

This document complements:
- `docs/lld/prediction_models/lld.md`
- `docs/lld/deep_learning/lld.md`
- `docs/lld/indicators/lld.md`
- `docs/lld/data_store/lld.md`

## 2. Problem
The current prediction runtime can load saved artifacts and print a normalized JSON prediction. That is useful for manual smoke tests, but it is not enough for an autonomous trading system.

Current gaps:
- runtime predictions are lost unless manually copied or redirected
- offline backtests do not measure day-by-day live prediction behavior
- the future AI/testnet agent needs historical model signals to explain why it bought, sold, or held
- there is no simple way to compare predicted return against the actual future return after the horizon passes
- there is no database-backed source for recent model quality, drift, or calibration analysis

The prediction journal fills this gap. It is not a training system and not an order ledger. It is the model-signal audit layer.

## 3. Scope
Included in scope:
- persist predictions generated by `PredictionRuntimeService`
- support tabular, statistical, and deep-learning artifacts
- store enough model, artifact, target, and signal metadata to replay the model decision context
- settle predictions after the target candle exists
- compute actual value, actual return, absolute error, percentage error, and direction correctness
- expose CLI commands for journaling, settlement, and summary reporting
- keep the design compatible with future scheduled per-minute inference

Out of scope:
- exchange order journal
- account, balance, or portfolio ledger
- LLM agent decision journal
- model registry/version table beyond artifact metadata required by the journal
- retraining orchestration
- feature-vector snapshot storage in the first implementation
- advanced experiment tracking systems such as MLflow or Weights & Biases

## 4. Design Drivers
The design is shaped by:
- the existing DB-first architecture
- the current normalized `RuntimePrediction` output
- the need to evaluate predictions after the horizon candle closes
- future testnet trading, where model signals and agent decisions must be auditable
- the requirement to avoid duplicate prediction rows when commands are retried
- the need to support multiple model types with one common journal shape

Pragmatic decisions:
- journaling is explicit at first, not default
- settlement is a separate command, not implicit in `predict`
- store artifact path and artifact hash so retrained models can be distinguished even if names repeat
- store feature names and metadata, but not the full feature vector initially
- keep prediction journal separate from future trading decision and order journals

## 6. Data Model
Recommended table name:

```text
prediction_journal
```

Recommended columns:

```sql
id BIGSERIAL PRIMARY KEY,
created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),

symbol TEXT NOT NULL,
interval TEXT NOT NULL,
model_name TEXT NOT NULL,
artifact_kind TEXT NOT NULL,
artifact_path TEXT NOT NULL,
artifact_sha256 TEXT,

reference_time TIMESTAMPTZ NOT NULL,
prediction_time TIMESTAMPTZ NOT NULL,
forecast_horizon INTEGER NOT NULL,
target_field TEXT NOT NULL,
target_mode TEXT NOT NULL,

reference_value DOUBLE PRECISION NOT NULL,
predicted_value DOUBLE PRECISION NOT NULL,
predicted_return DOUBLE PRECISION NOT NULL,
predicted_direction TEXT NOT NULL,

feature_names JSONB NOT NULL DEFAULT '[]'::jsonb,
metadata JSONB NOT NULL DEFAULT '{}'::jsonb,

actual_value DOUBLE PRECISION,
actual_return DOUBLE PRECISION,
actual_direction TEXT,
absolute_error DOUBLE PRECISION,
absolute_percentage_error DOUBLE PRECISION,
direction_correct BOOLEAN,
settled_at TIMESTAMPTZ
```

Direction values:
- `up`
- `down`
- `flat`

Direction derivation:
- predicted direction is derived from `predicted_return`
- actual direction is derived from `actual_return`
- `flat` is used only when the return is exactly zero unless a future epsilon is introduced

## 7. Constraints And Indexes
Recommended unique constraint:

```sql
UNIQUE (
  symbol,
  interval,
  model_name,
  artifact_sha256,
  reference_time,
  prediction_time
)
```

If `artifact_sha256` is unavailable, the implementation should either:
- compute it before writing, or
- fall back to `artifact_path` in the uniqueness rule

Recommended indexes:

```sql
CREATE INDEX prediction_journal_unsettled_idx
  ON prediction_journal (symbol, interval, prediction_time)
  WHERE settled_at IS NULL;

CREATE INDEX prediction_journal_model_time_idx
  ON prediction_journal (symbol, interval, model_name, reference_time DESC);

CREATE INDEX prediction_journal_created_idx
  ON prediction_journal (created_at DESC);
```

Reason:
- settlement needs fast access to unresolved rows whose prediction time has passed
- model analysis needs time-range scans by symbol, interval, and model
- recent operational inspection needs `created_at`

## 8. Journal Write Flow
Runtime journaling flow:

1. User or scheduler calls `capm predict`.
2. `PredictionRuntimeService` loads the artifact and creates a `RuntimePrediction`.
3. If journaling is enabled, `PredictionJournalService` converts `RuntimePrediction` into a journal row.
4. The service computes artifact SHA256 when an artifact path is available.
5. The repository inserts the journal row with idempotent conflict handling.
6. CLI prints the normal prediction JSON plus `journal_id` if a row was written.

Initial CLI:

```bash
uv run capm predict \
  --model-artifact experiments/results/<run_id>/model.pkl \
  --symbol BTCUSDT \
  --interval 1m \
  --journal
```

Default behavior:
- no journal write unless `--journal` is provided

Reason:
- this avoids surprising DB writes during manual experiments
- scheduled inference can opt in explicitly

## 9. Settlement Flow
Settlement is the process of filling actual outcome columns after the target candle exists.

Settlement flow:

1. Command selects unsettled rows where `prediction_time <= until`.
2. Repository checks whether the target candle exists for each row.
3. If no candle exists yet, leave the row unsettled.
4. If the candle exists, compute:
   - `actual_value`
   - `actual_return`
   - `actual_direction`
   - `absolute_error`
   - `absolute_percentage_error`
   - `direction_correct`
   - `settled_at`
5. Repository updates the journal row.
6. CLI prints a settlement summary.

Initial CLI:

```bash
uv run capm settle-predictions \
  --symbol BTCUSDT \
  --interval 1m \
  --until 2026-05-30T00:00:00Z
```

If `--until` is omitted:
- use current UTC time minus one interval as the conservative default

Reason:
- the target candle should be closed before settlement

## 10. Summary And Analytics Flow
The journal should support lightweight model-quality inspection without notebooks.

Initial CLI:

```bash
uv run capm prediction-journal summary \
  --symbol BTCUSDT \
  --interval 1m \
  --start 2026-05-01T00:00:00Z \
  --end 2026-05-30T00:00:00Z
```

Recommended summary output:
- count of predictions
- count of settled predictions
- MAPE
- RMSE
- direction accuracy
- mean predicted return
- mean actual return
- predicted-return quantiles
- actual-return quantiles
- counts by predicted direction
- counts by actual direction

Useful analysis queries:
- MAPE by model and day
- direction accuracy by model and day
- predicted return bucket versus actual return
- false-positive buy signal rate
- model drift over recent windows
- live performance versus offline backtest expectations

## 11. Repository Contract
Recommended repository methods:

```python
class PredictionJournalRepositoryPort(Protocol):
    def save_prediction_journal_entry(self, entry: PredictionJournalEntry) -> PredictionJournalEntry:
        ...

    def get_unsettled_prediction_journal_entries(
        self,
        symbol: str,
        interval: str,
        until: datetime,
        limit: int,
    ) -> tuple[PredictionJournalEntry, ...]:
        ...

    def settle_prediction_journal_entry(self, settlement: PredictionJournalSettlement) -> PredictionJournalEntry:
        ...

    def summarize_prediction_journal(
        self,
        symbol: str,
        interval: str,
        start_time: datetime,
        end_time: datetime,
        model_name: str | None = None,
    ) -> PredictionJournalSummary:
        ...
```

Implementation note:
- the first version can keep summary calculations in the service layer if repository-level aggregate SQL would slow implementation
- settlement should be idempotent
- duplicate prediction writes should return the existing row or update deterministic metadata, not create duplicates

## 12. Domain Entities
Recommended entities:

```python
PredictionJournalEntry
PredictionJournalSettlement
PredictionJournalSummary
```

`PredictionJournalEntry` should include:
- identity and timestamps
- model and artifact metadata
- reference and prediction timing
- predicted values and direction
- feature names and metadata
- optional settlement values

`PredictionJournalSettlement` should include:
- journal id
- actual value
- computed errors
- direction correctness
- settlement time

`PredictionJournalSummary` should include aggregate metrics and counts for CLI output.

## 14. Agent/Testnet Relevance
The prediction journal is the model-signal layer for the future demo/testnet trading system.

Expected later flow:

1. Latest closed candle is ingested.
2. Feature rows are computed.
3. Enabled model artifacts run predictions.
4. Each prediction is written to `prediction_journal`.
5. Agent decision logic reads the latest predictions and chooses buy, sell, or hold.
6. Future trading decision journal records the agent decision separately.
7. Prediction settlement measures model quality independent of the agent decision.
8. Order/trade journal measures execution and portfolio outcomes.

Important separation:
- prediction journal answers: "what did each model forecast?"
- decision journal answers: "what did the agent decide and why?"
- order journal answers: "what was sent to the exchange?"
- portfolio ledger answers: "what happened to balances and positions?"

## 15. Operational Considerations
Retention:
- keep prediction rows indefinitely during development
- add retention or partitioning only after volume becomes a real issue

Volume estimate:
- one model, one symbol, 1m interval creates about 1,440 rows per day
- five models create about 7,200 rows per day per symbol
- 50 symbols and five models create about 360,000 rows per day

Performance:
- inserts should be small and frequent during live inference
- settlement can run in batches
- summary queries should filter by symbol, interval, model, and time range

Failure behavior:
- prediction can still print JSON if journal write fails, but CLI should exit non-zero when `--journal` was explicitly requested and persistence fails
- settlement should report skipped rows where target candles are missing
- duplicate writes should be safe

Security:
- artifact paths and metadata must not include secrets
- no API keys or exchange credentials should be written to the journal

## 18. Open Decisions
- Should `capm predict` journal by default once the testnet scheduler exists?
- Should the unique key use artifact SHA256 only, or artifact path plus SHA256?
- Should we store a compact feature snapshot for model explainability?
- Should settlement use exact `prediction_time` candle open only, or tolerate missing target candle by using the next available candle?
- Should summary metrics live in SQL aggregates or Python service code?
- Should prediction rows be partitioned by month if multi-symbol inference creates high volume?

## 19. Known Limitations
- This design does not solve model quality.
- This design does not persist agent reasoning or exchange orders.
- First implementation should not store full feature vectors, so forensic replay is limited to artifact metadata and feature names.
- Prediction settlement depends on target candles already being ingested.
- Direction correctness is threshold-free at first; future signal-quality reports may need threshold-aware direction buckets.
